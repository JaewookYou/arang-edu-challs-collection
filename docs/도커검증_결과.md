## 1. 기동 — DB readiness 수정 라이브 확인

`up` 로그에서 의존 순서가 의도대로 동작함을 확인:

```
Container db  Healthy
Container sqli-1  Starting        ← db 가 healthy 된 뒤에야 기동
Container sqli-2  Starting
Container sqli-3  Starting
Container jsp-sqli Starting
```

`docker logs db` → `[db] init done` (chall DB·sqli 유저·테이블 생성 완료). 기동 직후
`http://localhost:9201` 등 sqli 문제 첫 요청부터 `db error` 없이 정상 응답.
→ "중급 문제가 중간에 망가진다"의 원인(단축형 `depends_on` 경쟁)이 healthcheck +
`condition: service_healthy` 로 **실제로 해소됨**.

## 2. 기동 중 발견·수정한 결함 (★ 빌드/런타임 — 이게 없으면 풀이 자체 불가)

| # | 파일 | 증상 | 원인 | 수정 |
|---|---|---|---|---|
| 1 | `bot/Dockerfile` | **빌드 실패** — `apt-get install chromium` 가 `libwayland-egl1`·`libz3-4` 에서 `400 Bad Request`. 봇 빌드 실패로 `up` 전체 중단(클라이언트 8문제 불능) | deb.debian.org(Fastly CDN) 으로의 apt **HTTP 파이프라이닝 + 스테일 캐시** 가 특정 .deb 에 400 반환(재시도해도 동일) | apt 설정 추가: `Pipeline-Depth "0"`(요청 1건씩) · `No-Cache "true"`(캐시 무효화) · `Retries "8"` → 빌드 안정화 |
| 2 | `challenges/jsp/jsp-pathtraversal/Dockerfile` | 컨테이너 **크래시 루프**(`flag.txt: No such file or directory` 무한 반복) | webapp 소스에 `WEB-INF/` 가 없어, 기동 시 `echo $FLAG > ROOT/WEB-INF/flag.txt` 의 상위 디렉터리가 부재 → 쓰기 실패로 즉시 종료 | CMD 에 `mkdir -p .../ROOT/WEB-INF` 선행 |
| 3 | `challenges/server/upload-webshell/Dockerfile` | 웹쉘 업로드는 되나 **실행 안 됨**(`.phtml` 이 PHP 소스 그대로 출력) → flag 획득 불가 | (a) uploads/ 가 첫 요청 전엔 없어 기동 시 `.htaccess` 생성 실패 (b) Debian Apache 기본 `AllowOverride None` 이라 `.htaccess` 자체가 무시됨 | `.htaccess` 대신 **서버설정(conf-enabled)에 SetHandler** 로 `.phtml`·`.php5` 전역 실행 매핑 + uploads/ 빌드시 생성 |
| 4 | `challenges/logic/race-transfer/Dockerfile` | 동시요청을 보내도 **항상 vault=800 에서 멈춰** flag(≥1000) 도달 불가(결정적) | 컨테이너가 **gunicorn `-w 1 --threads 8`** 로 기동 → 동시 검증 통과가 최대 8요청 → amt=100·me=100 로는 8×100=800 에서 막힘. (경합 자체는 정상 — me 가 음수로 오버드래프트됨) | 목표(1000)는 그대로 두고 `--threads 8 → 24` 로 키워 10요청 이상 동시 통과 가능하게 함 → **원본 목표 1000 으로 5/5 안정 풀이**. 비원자 검증/차감 로직(취약점)은 그대로 |
| 5 | JSP 전체(`challenges/jsp/**/*.jsp`·`jsp-chain/webapp/index.jsp`) + `jsp-upload/src/UploadServlet.java` + jsp-upload·jsp-chain Dockerfile | **JSP 페이지 한글 전부 깨짐** | Tomcat 이 JSP 응답을 charset 미지정 시 기본 ISO-8859-1 로 서빙 → UTF-8 한글 깨짐 | 모든 JSP 에 `<%@ page contentType="text/html; charset=UTF-8" pageEncoding="UTF-8" %>` 추가 · 서블릿 `setContentType("…charset=UTF-8")` · `javac -encoding UTF-8` → 4문제 모두 `charset=UTF-8` 정상 |
| 6 | `challenges/client/open-redirect/app.py` | 외부 도메인으로 **리다이렉트만 되고 flag 안 보임** | 성공 시 `302 + Location` 반환 → 브라우저가 그 사이트로 떠나 flag 본문을 못 봄 (curl 은 redirect 미추적이라 보였던 것) | 성공 시 **200 페이지로 flag 표시**(실제 이동은 안 함) → 브라우저에서도 flag 확인 |

> **외부 GitHub 챌린지 기동**: `authbypass-basic(:9001)`·`authbypass-advanced(:9005)`·`secret-tunnel(:8090, SSH 2222)` 는 각자 own docker-compose 라 메인 `start.ps1` 에 포함되지 않아 "접속 안 됨"으로 보였음. `start.ps1 -External` 스위치를 추가해 메인+외부를 함께 기동/정리하도록 함(먼저 `setup_external.ps1` 로 clone+플래그주입 필요). 3종 모두 기동·접속 확인.

## 4. 스코어보드 흐름 점검 (:9000)

- 회원가입 → 로그인 정상.
- `/submit` 에 **정답 flag** 제출 → `#ok`, 사용자 `solved` 목록 갱신(`/data/users.json` 에서 `['sqli-1','xss-1','jsp-chain']` 확인). **오답** → `#no`.
- `/ranking` 에 해결 수 반영. (플랫폼은 `registry.yaml` + 자신의 env `FLAG_*` 만으로 채점 — 문제 컨테이너와 직접 통신 안 함)

## 5. 클라이언트(봇) 검증 메모

- report 의 URL 은 **서비스명**(`http://xss-1:9101/...`) 으로 제출(봇이 컴포즈 DNS 로 도달). `localhost` 입력 시 봇이 자기 자신을 쳐서 실패.
- exfil 수신지는 봇 컨테이너에서 도달 가능한 `host.docker.internal:<port>`(호스트 리스너) 사용 — 본 검증은 호스트에 HTTP 수신기를 띄워 확인. (XS-Leak 은 `text/css` 응답하는 CSS 누출 서버를 별도 구동)
- 봇 동작 확인: `docker logs bot` 에 `[bot] chal=.. url=..` 기록.
